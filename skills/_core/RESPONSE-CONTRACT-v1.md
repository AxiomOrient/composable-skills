# Response Contract v1

## 목표

사용자가 30초 안에 읽고 다음 행동을 알 수 있는 응답을 출력한다.

---

## 핵심 원칙 3가지

**1. 결론부터**
첫 문장에서 "무슨 일이 있었고 지금 뭐가 중요한지"를 말한다. 과정 설명은 뒤에 온다.

**2. 구체적으로**
파일 이름, 숫자, 함수 이름을 쓴다. "문제가 있다"가 아니라 "auth.service.ts 32번째 줄에서 토큰이 만료돼도 200을 반환한다"처럼 쓴다.

**3. 짧게**
한 문장에 한 가지 주장만 넣는다. 섹션당 항목은 최대 3개다.

---

## 응답 구조

### 단독 실행 (스킬 1개)

```markdown
## 결과
[무슨 일이 있었는지 + 사용자에게 뭐가 중요한지] 1-2문장.

## 한 일
- [무엇을] [왜 했는지] (최대 3개)

## 근거
- [파일/명령/수치] — [이게 뭘 보여주는지] (최대 3개)

## 다음에 할 것  ← 없으면 섹션 전체 생략
- [동사로 시작하는 행동] — 왜 중요한지

## 질문  ← 없으면 섹션 전체 생략. 있으면 항상 맨 마지막
1. [질문]
```

### 합성 실행 (스킬 2개 이상)

```markdown
## 결과
[파이프라인 전체 결과 + 사용자에게 뭐가 중요한지] 1-2문장.

## 진행한 작업
- [단계]: [무엇을 했고 왜] (최대 3개)

## 근거
- [파일/명령/수치] — [이게 뭘 보여주는지] (최대 3개)

## 다음 단계  ← 없으면 섹션 전체 생략
- [동사로 시작하는 행동] — 왜 중요한지

## 질문  ← 없으면 섹션 전체 생략. 있으면 항상 맨 마지막
1. [질문]
```

---

## 섹션별 작성 규칙

### 결과 섹션
무슨 일이 있었는지 + 사용자에게 뭐가 중요한지를 1-2문장으로 말한다.

나쁜 예:
> "wf-check-full-review 실행이 완료되었습니다. 다섯 개 스킬이 순차적으로 실행되어 TARGET_SCOPE에 대한 종합 리뷰가 수행되었습니다."

좋은 예:
> "src/auth에서 문제 3개를 찾았다. 그 중 하나는 인증 우회 가능성이 있어서 지금 바로 고쳐야 한다."

### 발견 / 한 일 섹션
각 항목은 **이름 — 위치 — 왜 중요한가** 순으로 쓴다.

나쁜 예:
> "check-module-walls에서 LEAKED_ASSUMPTION이 탐지되었습니다. HARDENING_ACTION이 필요합니다."

좋은 예:
> "**토큰 만료 미처리** — `auth.service.ts:87` — 만료된 토큰으로도 API가 200을 반환한다. 인증 우회 가능."

### 근거 섹션
파일 경로나 명령만 적지 않는다. 그게 무엇을 보여주는지 함께 적는다.

나쁜 예:
> "src/auth/auth.service.ts"

좋은 예:
> "`src/auth/auth.service.ts:87` — 만료 토큰 검사 없이 바로 응답 반환"

### 다음에 할 것 섹션
"~을 확인한다", "~을 수정한다"처럼 동사로 시작하는 행동을 쓴다.
추상적인 목표가 아니라 바로 실행할 수 있는 것만 쓴다.

나쁜 예:
> "보안 강화가 필요합니다."

좋은 예:
> "`auth.service.ts:87` validateToken() 에 만료 시간 체크 추가 — 인증 우회 막는 데 필수"

## 절대 출력 금지

사용자 응답에서 아래 항목을 절대 쓰지 않는다.

| 금지 | 대신 |
|---|---|
| `NX-001`, `NX-###` | 행동 텍스트만 |
| `P0`, `P1`, `P2` | "지금 바로" / 표시 생략 |
| `stage payload`, `canonical queue` | 쓰지 않음 |
| `response_profile`, `profile_id` | 쓰지 않음 |
| `lens:`, `hickey-carmack`, `feynman`, `eisenhower` | 쓰지 않음 |
| `$skill`, `$wf-*`, `scope:diff` | 쓰지 않음 |
| `TASK-ID`, `DOING`, `BLOCKED` | 쓰지 않음 |
| `uncertain` | "확인 필요" |
| `KEY_FINDING`, `LEAKED_ASSUMPTION`, `HARDENING_ACTION` | 일반 단어로 |

## 특수 상황 처리

**근거가 없을 때**
추측하지 않는다. "확인 필요 — [가장 간단한 확인 방법]"을 적는다.

**문제가 없을 때**
억지로 문제를 만들지 않는다. "이 범위에서 문제를 찾지 못했다"라고 쓴다.

**질문이 있을 때**
응답 맨 마지막 "질문" 섹션에만 적는다. 다른 섹션 안에 질문을 넣지 않는다.

**종료 스킬 (commit, release GO)**
"다음에 할 것" 섹션을 생략한다. 작업이 완료된 것이다.

**clarify 포함 파이프라인**
질문 섹션은 항상 응답 최하단에만 둔다. 다른 섹션이 먼저 오고 질문이 마지막에 온다.

---

## Profile Switch Rule

단독 실행에서 `response_profile`이 있으면 `RESPONSE-PROFILES-v1.md`의 해당 프로필 섹션 순서를 따른다.
합성 실행(스킬 2개 이상)은 항상 합성 실행 기본 템플릿을 사용한다.
