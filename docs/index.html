Composable Skills

작업을 크게 만들지 않는다. 핵심을 나누고, 끝까지 간다.

Composable Skills는 한 스킬에 한 책임만 둡니다. 그래서 질문은 더 또렷해지고, 구현은 더 좁아지고, 검증은 더 분명해집니다. 단순함은 비어 있는 상태가 아니라, 중요한 것이 한 번에 보이는 상태입니다.

스킬 아틀라스 보기 설치와 배포

Philosophy

철학은 짧고 분명해야 한다.

이 시스템은 많은 기능을 한곳에 몰지 않습니다. 대신 책임을 잘게 나누고, 연결은 드러내고, 결과는 근거로 확인합니다.

01

단순함은 축소가 아니다

적게 넣는 것이 아니라, 핵심이 바로 보이게 만드는 것이다.

02

한 스킬, 한 책임

한 번에 하나의 문제만 맡기면 선택과 수정이 모두 쉬워진다.

03

확신보다 근거

좋아 보인다가 아니라, 왜 맞는지 확인 가능한 흔적을 남긴다.

What Users Get

사용자는 더 적게 고민하고, 더 정확하게 고른다.

이 시스템이 주는 가치는 도구 수가 아니라 판단 품질입니다. 무엇을 먼저 해야 하는지, 어디까지 하면 되는지, 무엇을 남겨야 하는지가 빨라집니다.

덜 헤맨다

workflow가 첫 진입점을 잡아 주기 때문에 시작 비용이 줄어든다.

덜 섞인다

atomic이 한 책임만 맡기 때문에 문제와 해결이 서로 엉키지 않는다.

덜 망가뜨린다

plan, build, test, review가 검증 근거를 남겨 불필요한 추측을 줄인다.

덜 남긴다

tidy, doc, release가 끝맺음을 맡아 레거시 대신 정리된 상태를 남긴다.

System

시스템은 간결해야 한다.

표면은 작게, 연결은 보이게, 반복은 통제 가능하게. 이 세 가지가 이 스킬 시스템의 구조입니다.

Workflow

한 덩어리 작업의 기본 진입점

질문 정리, 디버그, 계획 수립, 구현, 테스트 보강 같은 큰 흐름을 맡는다.

Atomic

정확히 하나의 동작만 수행하는 공구

분석, 계획, 구현, 테스트, 정리처럼 필요한 동작만 정밀하게 꺼내 쓴다.

Control

끝날 때까지 밀어 주는 제어 장치

루프 실행, 체인 조립, 최종 정리처럼 반복과 종료 조건을 관리한다.

작게 공개한다. workflow expansion은 숨기지 않는다. 근거 없이 완료라고 말하지 않는다.

Operating Flow

좋은 흐름은 순서가 분명하다.

문제를 정리하고, 구조를 보고, 계획을 세우고, 구현하고, 검증하고, 정리한다.

01

질문을 날카롭게 만든다

workflow-clarify-request, ask-*, clarify-*

02

현재 상태를 본다

workflow-scout-structure, analyze-*

03

만들기 전에 선을 긋는다

workflow-plan-build-ready, plan-*

04

구현과 수정을 좁게 한다

build-*, debug-*, workflow-build-*

05

보호막과 판정을 붙인다

test-*, review-*, release-*

06

군살 없이 마무리한다

tidy-*, doc-*, control-*

Install & Deploy

설치와 배포도 복잡할 이유가 없다.

런타임 프레임워크 없이 `docs/` 정적 파일로 배포한다. 카탈로그만 다시 만들면 사이트는 곧바로 최신 상태가 된다.

설치

git clone git@github.com:AxiomOrient/composable-skills.git
cd composable-skills
./scripts/sync.sh

카탈로그 갱신

python3 scripts/build_site_data.py

로컬 확인

python3 -m http.server 4173 --directory docs

GitHub Pages

저장소 설정에서 Pages 소스를 기본 브랜치 /docs 폴더로 지정하면 된다.

저장소 README 시스템 문서 스킬 가이드 skills/ 디렉터리
