# Agent Skill Guide

공개 스킬 체계, 스킬 시스템 문서, 시스템 `skill-creator`를 기준으로 압축한 스킬 제작 가이드다.

## 1. 계약이 먼저다

스킬은 "좋게 해줘"를 받는 도구가 아니다. 하나의 job, 하나의 출력, 하나의 종료 조건이 먼저 있어야 한다.

- 나쁜 요청: `인증 시스템을 만들어 줘`, `코드에서 버그를 찾아줘`
- 좋은 요청: `bcrypt-12와 7일 만료 리프레시 토큰을 쓰는 JWT 인증을 구현해`, `데이터베이스를 훑고 관찰한 사실만 객관적으로 보고해`

좋은 스킬은 항상 좁은 `Purpose`, 명시적 `Required Inputs`, 재사용 가능한 `Structured Outputs`, 분명한 `Neutrality Rules`, 끝내는 검증 계약을 가진다.

## 2. 조사와 구현은 나눈다

구현 방법이 불명확하면 한 세션에 다 넣지 않는다. 같은 주제라도 실제 행위가 다르면 계약도 나눠야 한다.

- 조사용 스킬: 관찰, 옵션 비교, 범위 명확화, 기준값 측정
- 설계용 스킬: 요구 정리, 구조 설계, 검증 순서, 의존성 규칙
- 테스트용 스킬: 케이스 설계, 갭 점검, 사용자 시나리오 검증
- 구현용 스킬: 코드 변경, 성능 개선, 완료 조건까지의 반복

긴 세션 하나로 밀어붙이면 문맥이 비대해진다. 새 작업은 가능한 한 새 계약과 새 문맥으로 시작한다.

## 3. 중립성은 기본값이다

에이전트는 사용자를 만족시키려는 편향이 있으므로 결과를 유도하는 프롬프트를 피해야 한다.

- evidence와 inference를 분리한다
- 근거가 부족하면 `no finding`이나 `inconclusive`를 허용한다
- 없는 결함이나 과장된 결론을 만들지 않는다

분석, 리뷰, 릴리즈 판정 계열은 이 규칙이 특히 강해야 한다.

## 4. 종료 조건이 없으면 중간에 멈춘다

스킬에는 반드시 "언제 끝나는가"가 들어가야 한다. 좋은 종료 조건은 테스트 통과, 스크린샷 검증, 계약 문서 충족, 릴리즈 체크 통과처럼 확인 가능한 신호다.

## 5. Lens는 페르소나가 아니라 멘탈 모델이다

이 저장소의 Lens는 roleplay가 아니라 사고 규칙 카드다. 세계적인 석학이나 실무가 이름을 붙여도 핵심은 아래 다섯 가지다.

- `summary`
- `thesis`
- `decision_rules`
- `anti_patterns`
- `required_artifacts`

예:

- `kahneman-tversky`: 관찰과 해석을 분리한다
- `hickey-carmack`: 데이터, 경계, 비용을 먼저 드러낸다
- `feynman-teaching`: 쉬운 말과 예시로 먼저 설명한다

즉, Lens는 멋이 아니라 사고 편향을 통제하는 장치여야 한다.

## 6. SKILL.md는 짧게, 자세한 것은 밖으로 뺀다

기본 원칙은 "Codex는 이미 똑똑하다"이다. SKILL.md에는 비자명한 절차만 남기고 자세한 것은 분리한다.

- 반복 코드: `scripts/`
- 자세한 지식: `references/`
- 결과물 템플릿: `assets/`

README, CHANGELOG 같은 부가 문서는 스킬 폴더에 넣지 않는다.

## 7. 새 스킬보다 기존 경계를 먼저 본다

새 스킬을 만들기 전에 먼저 묻는다.

- 기존 스킬 입력을 더 명확히 쓰면 해결되는가
- 새 스킬이 아니라 기존 스킬의 이름, prefix, 설명을 바로잡으면 해결되는가
- atomic으로 충분한가
- 반복 조합이면 workflow가 맞는가
- 직접 문제를 풀지 않으면 utility가 맞는가

반복 조합이 정말 고정될 때만 `wf-*`를 만든다. 경계 문제라면 새 폴더를 늘리기 전에 리네임이나 재분류부터 검토한다. 그 외에는 기존 스킬 합성이 더 단순하다.

## 8. 규칙과 스킬은 주기적으로 정리한다

규칙은 금지사항을, 스킬은 수행 방식을 고정한다. 둘 다 계속 늘어나면 충돌과 문맥 비용이 커진다.

- 겹치는 스킬
- 레거시 workflow
- 과한 규칙
- 구현, 레지스트리, 가이드 문서 사이의 이름·카테고리 불일치
- 안 쓰는 문서와 예시

특히 prefix, 예시, 개수처럼 쉽게 낡는 정보는 고정 수치보다 구조 설명을 우선한다.

한 줄 요약:

좋은 스킬은 많이 설명하는 스킬이 아니라, 좁은 계약과 중립적 사고, 명확한 종료 조건, 작은 문맥으로 반복 작업을 안정화하는 스킬이다.
